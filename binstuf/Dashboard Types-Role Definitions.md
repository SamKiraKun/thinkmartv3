## Prompt: ThinkMart Multi-Dashboard RBAC + Revenue-Sharing Specification (Agent-Strict)

You are a **code-focused engineering agent** implementing and validating **role-separated dashboards** and **RBAC enforcement** for the ThinkMart platform. Your output must be **implementation-ready** (code, structure, routes, guards, middleware, tests, and documentation updates as applicable), based strictly on the provided codebase/stack.

### 0) Primary Goal

Implement a system where **dashboards are strictly separated by role**, **not shared**, and **cannot be accessed interchangeably**. Every role must only see and mutate data it is allowed to access. This applies to:

* UI routes/pages (frontend)
* API routes/endpoints (backend)
* Database queries (data access layer)
* Business logic services (domain rules)
* Logs/audits (traceability)

**Definition of “strict separation”**:
A user authenticated as Role A must be technically unable (not just “not shown in UI”) to:

* load Role B dashboard pages,
* call Role B APIs,
* read Role B scoped data,
* execute Role B actions.

# 1) Dashboards (Authoritative Role Definitions)

The platform has the following dashboards. Treat this section as the **single source of truth**.

## 1.1 User Dashboard (End User)

**Purpose:** marketplace usage + earning system.

**Allowed actions:**

* Browse products
* Complete tasks (if present in project)
* Earn rewards
* Place orders
* Withdraw earnings

**Hard restrictions:**

* No administrative actions
* No city partner analytics
* No vendor management
* No school/org management
* No system settings access

---

## 1.2 Partner Dashboard (City Investors)

**Identity:** Partner = investor assigned to **one specific city**.

**Key constraints:**

* Multiple partners can exist in the same city.
* Each partner can have a **revenue share percentage**, controlled by Admin.
* Each partner’s share is **<= 20%** (cap per partner).
* Partner visibility is limited to their assigned city.

### Partner revenue logic (must be implemented exactly)

When a **user withdraws ₹X**:

* Up to **20% of ₹X** is paid **by the company** to the partner(s) of that city.
* This payout is **NOT deducted from the user’s earnings**.
* The user receives the full ₹X (assuming withdrawal succeeds).
* Partner payout happens as part of the withdrawal flow (atomicity requirements below).

**Partner payout distribution rule:**

* The **exact partner percentage(s)** are defined by Admin.
* If multiple partners exist in the same city, payout must follow the configured percentages per partner.
* If no partners exist for a city, payout is ₹0.

**Partner dashboard can see:**

* Earnings and analytics **only for their city scope**
* Their own wallet balance and payout history
* Aggregated metrics for that city (only those needed for partner insights)

**Partner dashboard cannot:**

* View users outside their city
* Modify products/orders
* Modify commissions/percentages
* Access admin or vendor tools

## 1.3 Vendor Dashboard (Marketplace Seller)

**Purpose:** Manage product catalog and process orders (ThinkMart marketplace seller view).

**Vendor can:**

* Add/update/remove products
* Manage pricing, inventory, availability
* View and process orders placed by users

**Vendor restrictions:**

* Vendors can only manage **their own** catalog/products
* Vendors can only view orders that include **their products**
* No ability to view platform-wide earnings, partners, or system configs

## 1.5 Admin Dashboard (God-Level)

**Purpose:** Full control over platform rules and entities.

Admin can manage:

* Users

* Schools/orgs (even if planned, admin may configure future entities)

**Admin is the only role** that can:

**Identity:** Sub-admin is assigned by Admin with **limited and configurable permissions**.

**Hard constraints:**

* Sub-admin is NOT a full admin.
* Their permissions are explicitly configured by Admin.
* Every sub-admin action must be permission-checked.

# 2) Non-Negotiable Rules

## 2.1 RBAC Must Be Enforced Server-Side

UI hiding is not security. Enforce role checks in:

* backend middleware/guards
* service layer (defense in depth)
* data access filters

## 2.2 No Cross-Dashboard Route Access

If a user attempts to access a dashboard route not matching their role:

* deny access (401/403)
* do not leak role data
* redirect to correct dashboard where appropriate (frontend)

## 2.3 Scope Enforcement (City, Vendor Ownership)

Some roles require additional scoping:

* Partner → city scope enforced in every query
* Vendor → vendor ownership scope enforced in every query
* Sub-admin → permission set enforced per action

## 2.4 Withdrawal Flow Must Be Consistent + Safe

The withdrawal operation must:

* pay the user correctly,
* compute partner payout correctly,
* log transactions clearly,
* avoid double-paying (idempotency),
* avoid partial state updates (atomicity).

If DB supports transactions, use them. If not, implement compensating safety patterns.

# 3) Implementation Requirements (What You Must Produce)

## 3.1 Access Model

Implement roles:

* USER
* PARTNER
* VENDOR
* ADMIN
* SUB_ADMIN

Plus any needed scopes:

* partnerCityId (or city identifier)
* vendorId
* subAdminPermissions (list/bitmask/etc. depending on codebase)

## 3.2 Routing

Create/ensure distinct route namespaces, e.g.:

* `/dashboard/user/*`
* `/dashboard/partner/*`
* `/dashboard/vendor/*`
* `/dashboard/admin/*`
* `/dashboard/sub-admin/*`

And corresponding API namespaces, e.g.:

* `/api/user/*`
* `/api/partner/*`
* `/api/vendor/*`
* `/api/admin/*`
* `/api/sub-admin/*`

(Use conventions that match the existing codebase, but keep separation strict.)

## 3.3 Middleware / Guards

Create reusable guards such as:

* `requireAuth`
* `requireRole(role | roles[])`
* `requirePermission(permission)`
* `requireCityScope`
* `requireVendorOwnership`

Guards must return consistent errors and be used everywhere.

## 3.4 Partner Revenue Share Implementation

Implement payout logic for withdrawals:

* Determine user city
* Fetch partner(s) for that city
* For each partner: payout = withdrawalAmount * configuredPercentage
* Ensure percentage cap per partner (<= 20%)
* Ensure payout does not reduce user amount
* Record payout ledger entries

**Important:** Admin controls partner percentages; partners cannot modify them.

## 3.5 Audit Logging

Log at minimum:

* role
* actor user id
* target entity id (if any)
* action type
* timestamp
* before/after values for admin/sub-admin actions (where safe)

## 3.6 Tests (Mandatory)

Add tests that prove:

* A USER cannot access partner/vendor/admin/sub-admin APIs/routes
* A PARTNER cannot access user/admin/vendor data outside their city
* A VENDOR cannot manage products not owned by them
* SUB_ADMIN permissions are enforced per action
* Withdrawal triggers correct partner payout and is not deducted from user
* Double submission does not double pay (idempotency test)

Use the project’s existing test framework; if none exists, add minimal infrastructure consistent with the stack.

# 4) Acceptance Criteria (Must Be Verifiable)

## Phase A — RBAC Foundation

* All roles exist and are assigned deterministically.
* All protected routes return 401/403 consistently when unauthorized.
* Role mismatch cannot access other dashboards (UI + API).

## Phase B — Scope Enforcement

* Partner queries are city-scoped everywhere (no leaks).
* Vendor queries are ownership-scoped everywhere.
* Sub-admin permissions deny unauthorized actions.

## Phase C — Withdrawal + Partner Payout

* For a user withdrawal of ₹X:

  * user receives ₹X (assuming success),
  * partner(s) receive payouts based on configured percentages,
  * payout is not deducted from user,
  * payout is ₹0 if no partners exist for that city.
* Idempotency prevents double payout.

## Phase D — Hardening

* All privileged actions are logged.
* All list endpoints use pagination/limits if applicable.
* No sensitive fields are exposed accidentally in partner/vendor responses.

# 6) Edge Cases You Must Handle Explicitly

* User has no city assigned → withdrawal behavior must be defined (fail safely or default city rules; follow existing system logic).
* Multiple partners in a city → payout split must match configured percentages.
* Partner percentage misconfigured (>20%) → must be rejected at admin update time and/or clamped with warnings (choose one, justify).
* Rounding of payouts (paise/cents) → define deterministic rounding (e.g., round down to nearest integer) and keep consistent.
* Partial failures → no partial payout state; implement safe rollback/compensation.
* Planned School/Org dashboard code (if present) must not be reachable by non-admin.
